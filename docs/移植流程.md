# PicoC 跨平台移植流程总结

## 1. 目标先收敛
在移植 PicoC 之前，先把目标平台的产品形态定死，否则后面会在入口、标准库、文件系统、调试能力上反复返工。

建议先明确 4 件事：
- 运行方式：REPL、整文件执行、还是纯内嵌脚本接口
- 输入来源：串口、USB CDC、网络、文件系统、Flash、上位机下载
- 标准库范围：最小可用集，还是接近桌面版
- 错误恢复策略：脚本报错后回控制台，还是直接终止当前宿主任务

本工程落地后的结论是：
- 默认 REPL
- 支持串口整文件执行
- 不依赖文件系统
- 先保留 mini-stdlib

这个结论对后续换平台也适用：先收敛产品边界，再开始代码移植。

## 2. 先分清 3 层结构
PicoC 的移植最好拆成 3 层，而不是一开始就改解释器内核。

### 2.1 内核层
这是 PicoC 自身的语法、词法、类型、变量、表达式、解析和堆管理代码，例如：
- `lex.c`
- `parse.c`
- `expression.c`
- `type.c`
- `variable.c`
- `heap.c`

这层尽量少改，只做编译器兼容修正和必要 bug 修正。

### 2.2 平台层
这是 PicoC 和宿主环境的边界，重点是：
- 输出字符
- 获取字符/获取一行
- 退出脚本
- 文件读取
- 平台库注册

典型接口是：
- `PlatformPutc()`
- `PlatformGetCharacter()`
- `PlatformGetLine()`
- `PlatformExit()`
- `PlatformReadFile()`
- `PicocPlatformScanFile()`
- `PlatformLibraryInit()`

### 2.3 应用适配层
这是你自己的业务层，不属于 PicoC 上游，负责：
- 创建 PicoC 实例
- 驱动 REPL
- 做控制命令
- 组织脚本上传和执行流程
- 管理提示符、状态机、异常恢复

本工程对应的是：
- `picoc_app.c`
- `serial_app.c`
- `main.c`

换平台时，最应该复用的是这套分层思路，而不是照搬某个文件名。

## 3. 平台移植的推荐顺序
### 3.1 第一步：先让最小内核编过
先把 PicoC 核心源码编进目标工程，只追求：
- 编译通过
- 链接通过
- 不追求 REPL 好用

这一步要优先处理：
- 编译器兼容问题
- 宏定义问题
- 堆和栈策略
- `setjmp/longjmp` 是否可用

经验上，不要一上来就接文件系统、标准库、调试器。

### 3.2 第二步：只打通单向输出
先实现 `PlatformPutc()`，让 PicoC 可以往外打印。

只要打印通了，就能先看到：
- 启动信息
- 语法错误
- 运行错误
- 调试输出

这一步可以极大减少后续定位成本。

### 3.3 第三步：打通最小输入
先实现：
- `PlatformGetCharacter()`
- `PlatformGetLine()`

但不要立刻依赖 PicoC 原生交互入口。很多嵌入式平台更适合自己做一层输入状态机，再把完整源码交给 `PicocParse()`。

实践证明，嵌入式上“自己控输入流，PicoC 只负责解释执行”更稳定。

### 3.4 第四步：确定脚本执行入口
这一层最关键，建议只在两种入口里选一种或两种：
- 行交互 REPL：适合调试
- 整段源码执行：适合上传整份程序

不建议一开始就依赖桌面版那套 `main(argc, argv)` + 文件系统执行模型。

本工程的做法是：
- REPL 走 `PicocParse()`
- 整文件上传也走 `PicocParse()`
- 如果上传内容新定义了 `main()`，再调 `PicocCallMain()`

这套思路可直接迁移到其它 MCU、Linux 串口代理、USB 设备或 RTOS shell。

## 4. 入口设计原则
换平台时，不建议把上游 `picoc.c` 当作最终入口照搬。

更稳妥的方式是：
- 由目标平台自己的 `main()` 或任务入口驱动 PicoC 生命周期
- PicoC 只作为一个被调用的组件

推荐入口模型：
1. 宿主初始化硬件/系统资源
2. 初始化输入输出通道
3. 初始化 PicoC
4. 进入主循环或任务循环
5. 周期性驱动 PicoC 适配层

这样后面切换到：
- 不同 MCU
- RTOS 任务模型
- USB 或 TCP 控制台
- Bootloader 环境

都不需要重做解释器生命周期。

## 5. 标准库移植策略
### 5.1 不要一开始追求完整 libc
完整标准库移植成本高，而且嵌入式平台很多函数没有实际意义，比如：
- 文件 IO
- 进程控制
- 环境变量
- 时间/时区
- shell 命令执行

更实际的做法是分 3 档：

#### 档位 A：最小集
目标是尽快跑通解释器，只保留：
- `printf`
- `getchar` / `gets`
- `malloc/calloc/free`
- `strlen/memcpy/memset`

#### 档位 B：常用头注册
逐步补齐常见头名：
- `stdio.h`
- `stdlib.h`
- `string.h`
- `stdbool.h`

即使底层不是完整桌面实现，也先让 `#include` 不报错。

#### 档位 C：平台自定义头
暴露板级 API，例如：
- `picoc_stm32.h`
- `picoc_board.h`
- `picoc_hal.h`

这类头应该是后续扩展脚本能力的主要入口。

### 5.2 推荐做法
优先复用 PicoC 现有 mini-stdlib 能力，再在平台层注册头文件名。

这比直接搬完整 `cstdlib/*.c` 更稳，因为：
- 依赖更少
- 宿主假设更少
- 在小系统上更容易控风险

## 6. 文件系统策略
移植时一定要尽早决定：要不要支持真实文件系统。

### 6.1 如果不要文件系统
那就明确：
- `PlatformReadFile()` 返回不支持
- `PicocPlatformScanFile()` 返回不支持
- 只通过串口、内存、Flash、网络缓冲提交源码

这能大幅降低复杂度。

### 6.2 如果要文件系统
那就单独实现：
- 源码读取
- include 查找路径
- 错误提示中的文件名定位
- 缓存与内存回收

不要把这一步和最小 REPL 跑通绑在一起。

## 7. 控制台协议建议
不管换到什么平台，建议都单独设计一层控制台协议，而不是完全依赖 PicoC 原生交互逻辑。

推荐保留两种模式：
- `REPL` 模式：逐行交互
- `LOAD` 模式：批量接收源码

推荐命令：
- `:load`
- `:end`
- `:abort`

这样做的好处：
- 容易对接上位机
- 容易支持整文件发送
- 容易做错误恢复
- 容易切换输入通道

这部分是最有复用价值的移植经验。

## 8. 调试能力的判断原则
PicoC 源码里虽然有 debugger 框架，但移植时不要默认它已经是可用产品功能。

判断标准要分开：
- 内核里有没有断点表和调试状态
- 宿主入口有没有启用 debugger
- 用户有没有可操作的断点/继续/单步命令

如果最后两项没有，就不要把“支持断点调试”写进移植目标。

经验结论：
- 可以保留调试器框架
- 但不建议把“完整断点/单步系统”作为前期重点

## 9. 错误恢复原则
跨平台移植时，错误恢复策略要早定。

建议：
- 脚本错误通过 `PlatformExit()` 回到宿主控制流
- 不让脚本错误直接把整个平台主循环拖死
- 只有底层硬件初始化失败才进入宿主的致命错误处理

这套策略对 MCU、RTOS、PC 工具端都成立。

## 10. 推荐验证路径
每换一个新平台，都按固定顺序验证，不要一上来测复杂脚本。

### 阶段 1：启动
- 上电/启动后能看到欢迎信息
- 提示符正确

### 阶段 2：单行表达式
- `1+2`
- 简单变量赋值

### 阶段 3：多行代码块
- `if/while/for`
- 函数定义

### 阶段 4：整段源码执行
- 批量发送程序
- 自动 `main()` 调用

### 阶段 5：错误恢复
- 语法错误
- 未定义符号
- 缓冲区溢出
- 重复上传

### 阶段 6：资源边界
- 长脚本
- 深递归
- 高频输入
- 重复多轮执行

## 11. 后续换平台时最该复用什么
真正可复用的不是某个 STM32 文件，而是下面这些决策和结构：

- 宿主主入口驱动 PicoC，而不是反过来
- 平台层只做 IO、退出、文件、平台库注册
- 应用层自己管理 REPL/LOAD 状态机
- 先最小可用，再逐步扩展标准头
- 文件系统和调试器晚一点再做
- 错误恢复统一走 `setjmp/longjmp`

如果后面切换到其它平台，例如：
- 其它 STM32
- GD32 / NXP / ESP32
- RTOS shell
- USB 虚拟串口
- PC 代理程序

都建议继续沿用这套流程。
