# PicoC 最佳移植方法总结

这份文档不是简单记录“STM32H750 上怎么改过一次”，而是总结一套更适合 MCU/RTOS/串口控制台场景的 PicoC 移植方法。  
目标是两件事：

- 先用最小代价把解释器稳定跑起来
- 再用清晰分层把 REPL、脚本上传、调试能力逐步扩进去

如果后续切换到其它 STM32、GD32、ESP32、NXP、RTOS shell、USB CDC 或 PC 代理程序，优先复用的应该是这套方法，而不是某几个具体文件。

## 1. 先定产品边界，再动代码

PicoC 移植最容易返工的地方，不是编译报错，而是产品形态一开始没收敛。

开始前先明确 4 个问题：

- 运行方式是什么：`REPL`、整文件执行、还是纯内嵌脚本接口
- 输入来源是什么：串口、USB、网络、Flash、文件系统、上位机下载
- 标准库范围是什么：最小可用集，还是接近桌面版
- 错误恢复策略是什么：脚本报错后回控制台，还是终止当前宿主任务

本工程最终选择的是：

- 默认 `REPL`
- 支持串口整文件上传执行
- 不依赖目标端文件系统
- 优先保留 `mini-stdlib`
- 脚本异常退出后回到宿主主循环

这一步越清楚，后面越不会在入口、文件系统、标准库和调试器之间反复返工。

## 2. 最好的分层方式是“三层解耦”

最稳妥的移植方式，不是直接修改 PicoC 内核，而是把工程拆成 3 层。

### 2.1 内核层

这一层是 PicoC 自身解释器逻辑，例如：

- `lex.c`
- `parse.c`
- `expression.c`
- `type.c`
- `variable.c`
- `heap.c`

原则：

- 尽量少改
- 只做编译兼容修正和必要 bug 修复
- 不把宿主业务逻辑塞进内核

### 2.2 平台层

这一层只负责 PicoC 和宿主环境之间的边界接口，典型内容包括：

- 输出字符
- 获取字符或一行输入
- 脚本退出
- 文件读取
- 平台库注册

关键接口通常是：

- `PlatformPutc()`
- `PlatformGetCharacter()`
- `PlatformGetLine()`
- `PlatformExit()`
- `PlatformReadFile()`
- `PicocPlatformScanFile()`
- `PlatformLibraryInit()`

原则：

- 只做平台桥接，不做 REPL 协议和业务状态机
- 所有“和硬件/系统耦合”的东西都留在这里

### 2.3 应用适配层

这一层是移植里最有价值的部分，负责：

- 创建和初始化 PicoC 实例
- 驱动 `REPL`
- 管理脚本上传和执行流程
- 处理控制命令
- 管理提示符、状态机和异常恢复
- 桥接调试协议和上位机

本工程对应的关键实现是：

- [Core/Src/picoc_app.c](F:\work\work\PICO\h750\UART_DMA_H750\Core\Src\picoc_app.c:1)
- [Core/Src/main.c](F:\work\work\PICO\h750\UART_DMA_H750\Core\Src\main.c:69)
- 串口收发适配层

结论很明确：

- 内核层尽量不动
- 平台层只做宿主边界
- 应用层负责交互和控制流程

这是 PicoC 移植里最推荐、也最容易复用的结构。

## 3. 最佳移植顺序

最好的做法不是一次把“解释器 + 标准库 + 文件系统 + 调试器”全接上，而是按固定顺序推进。

### 3.1 第一步：先让最小内核编译通过

目标只有两个：

- 编译通过
- 链接通过

这一阶段优先解决：

- 编译器兼容问题
- 宏定义和平台配置问题
- 堆/栈策略
- `setjmp/longjmp` 是否可用

这时不要急着做：

- 文件系统
- 完整标准库
- 调试器
- 复杂交互界面

### 3.2 第二步：先打通单向输出

优先实现 `PlatformPutc()`，只让 PicoC 能往外打印。

原因很简单：

- 有输出，才能看到启动提示
- 有输出，才能看到语法错误和运行错误
- 有输出，后面调试和定位成本会大幅下降

经验上，先通输出，远比先通输入更重要。

### 3.3 第三步：打通最小输入

再实现：

- `PlatformGetCharacter()`
- `PlatformGetLine()`

但不建议直接完全依赖 PicoC 原生交互入口。  
对 MCU 来说，更稳的方案通常是：

- 宿主自己收字符
- 宿主自己维护输入缓冲和状态机
- 拼出完整源码后再交给 `PicocParse()`

这也是本工程采用的方式：解释器只负责“解析和执行”，而不是直接控制整条输入链路。

### 3.4 第四步：确定统一执行入口

真正稳定的移植，一定要尽早统一“脚本如何进入解释器”。

推荐只保留两类入口：

- 行交互 `REPL`
- 整段源码执行

本工程的推荐方案是：

- `REPL` 输入走 `PicocParse()`
- 整文件上传也走 `PicocParse()`
- 如果上传源码新定义了 `main()`，再调用 `PicocCallMain()`

这样一来：

- `REPL` 和整文件共用一套解析执行逻辑
- 宿主更容易控制生命周期
- 更容易迁移到 USB、TCP、RTOS shell 等其它输入通道

## 4. 入口设计的最佳实践

不要把上游 `picoc.c` 直接当成目标平台最终入口。  
最推荐的模型是：宿主驱动 PicoC，PicoC 只是一个组件。

推荐入口模型：

1. 宿主初始化硬件和系统资源
2. 初始化串口、USB 或其它输入输出通道
3. 初始化 PicoC 实例
4. 进入主循环或任务循环
5. 在循环中周期性驱动 PicoC 应用适配层

这种结构最大的好处是，后面无论切换到：

- 裸机主循环
- RTOS 任务
- USB 虚拟串口
- TCP 控制台
- Bootloader 环境

都不需要重做解释器生命周期管理。

## 5. 标准库移植的最佳策略

### 5.1 不要一开始追求完整 libc

嵌入式上最常见的错误路线，是一开始就试图把完整标准库一次性补全。  
这通常投入大、收益低，而且很多接口在 MCU 上根本没意义，比如：

- 文件 IO
- 进程控制
- 环境变量
- shell 命令执行

### 5.2 推荐按 3 档推进

#### 档位 A：最小可运行集

先满足解释器跑起来，优先保留：

- `printf`
- `getchar` / `gets`
- `malloc/calloc/free`
- `strlen/memcpy/memset`

#### 档位 B：常用头注册

逐步补齐常见头名，让脚本侧 `#include` 不先报错：

- `stdio.h`
- `stdlib.h`
- `string.h`
- `stdbool.h`

#### 档位 C：平台扩展头

当解释器稳定后，再逐步暴露板级能力，例如：

- `picoc_stm32.h`
- `picoc_board.h`
- `picoc_hal.h`

这种自定义头，才是后续扩展脚本能力的主要入口。

### 5.3 经验结论

优先复用 PicoC 自带 `mini-stdlib`，再在平台层注册需要的头文件名。  
这是小系统上最稳、也最容易控风险的方式。

## 6. 文件系统策略要尽早定

### 6.1 如果当前不依赖文件系统

那就明确做成“不支持”，不要中间态。

建议：

- `PlatformReadFile()` 返回不支持
- `PicocPlatformScanFile()` 返回不支持
- 源码只从串口、内存、Flash 或网络缓冲进入

这样复杂度会明显下降。

### 6.2 如果后面需要文件系统

那就把它当成后续独立能力扩展，而不是最小移植的一部分。  
需要单独设计的内容包括：

- 源码读取
- include 查找路径
- 错误提示里的文件名定位
- 缓存和内存回收

结论：

- 最小移植先不绑文件系统
- 文件系统晚一点接，整体更稳

## 7. 最推荐的控制台协议做法

嵌入式移植时，不建议完全依赖 PicoC 原生交互逻辑。  
更好的方法是宿主自己定义一层控制台协议。

推荐保留两种模式：

- `REPL` 模式：逐行交互
- `LOAD` 模式：批量接收源码

推荐基础控制命令：

- `:load`
- `:end`
- `:abort`

如果需要继续扩展调试，再增加：

- `:bkpt`
- `:bkptclear`
- `:cont`
- `:step`
- `:eval`
- `:vars`
- `:set`

这套方式的好处是：

- 容易对接上位机
- 容易支持整文件发送
- 容易做错误恢复
- 容易切换输入通道
- 容易叠加调试协议

这是本工程最值得复用的移植经验之一。

## 8. 调试能力的最佳接入方式

PicoC 自带 `debug.c` 框架，但“源码里有调试器”不等于“产品已经支持调试”。  
真正要算支持调试，至少要打通 3 件事：

- 内核里有断点表和调试状态
- 宿主入口真的启用了调试流程
- 用户能通过串口或上位机操作断点、单步、继续和查看状态

### 8.1 当前本工程已落地的能力

- 断点设置与清除
- 单步执行
- 继续执行
- 断点停下后表达式求值
- 当前作用域变量枚举
- 断点停下后变量修改
- `Watch` 监视变量变化

### 8.2 推荐接入方式

- 调试器核心继续沿用 PicoC 原生 `debug.c`
- 在应用适配层桥接调试命令
- 整文件执行时，把主实例断点复制到隔离执行实例
- 上位机通过结构化串口消息驱动 UI

### 8.3 当前边界

- 主要面向上传文件执行，不是 REPL 历史命令调试
- 断点文件名当前按 `serial_load` 处理
- 断点在单次文件运行结束后自动清空
- `:eval` 当前更适合整数表达式
- `:set` 当前主要覆盖基础标量变量
- `Watch` 在停点时刷新，不是运行中的连续采样

经验结论：

- 调试器框架可以早保留
- 但调试功能建议在最小解释器跑通后再接入

## 9. 错误恢复的最佳策略

错误恢复一定要在移植早期就想清楚。  
对嵌入式平台来说，最好的原则通常是：

- 脚本错误退出，不拖死宿主主循环
- 解释器异常结束后能回到控制台
- 只有底层硬件初始化失败才进入宿主致命错误处理

本工程采用的关键方法是：

- 通过 `PlatformExit()` 结束当前脚本
- 基于 `setjmp/longjmp` 回到宿主控制流

这套方式对 MCU、RTOS、PC 工具端都成立。

## 10. 最推荐的验证路径

每换一个新平台，都建议按固定阶段验证，不要一上来就测复杂脚本。

### 阶段 1：启动验证

- 上电后能看到欢迎信息
- 提示符正确显示

### 阶段 2：单行表达式验证

- `1+2`
- 简单变量赋值

### 阶段 3：多行代码块验证

- `if/while/for`
- 函数定义

### 阶段 4：整段源码执行验证

- 批量发送程序
- 自动 `main()` 调用

### 阶段 5：错误恢复验证

- 语法错误
- 未定义符号
- 缓冲区溢出
- 重复上传
- 用户主动中止

### 阶段 6：资源边界验证

- 长脚本
- 深递归
- 高频输入
- 重复多轮执行

### 阶段 7：调试能力验证

- 断点设置和清除
- 单步执行
- 表达式求值
- 变量查看
- 变量修改
- `Watch` 变化刷新

## 11. 最该复用的不是代码，而是决策

真正可复用的，不是某个 STM32 文件，而是下面这些决策：

- 宿主主入口驱动 PicoC，而不是反过来
- 平台层只做 IO、退出、文件、平台库注册
- 应用层自己管理 `REPL/LOAD` 状态机
- 统一用 `PicocParse()` 作为主要执行入口
- 先最小可用，再逐步扩展标准头
- 文件系统和高级调试能力后置
- 错误恢复统一走 `setjmp/longjmp`
- 控制台和调试协议由宿主统一管理

后续如果切换到：

- 其它 STM32
- GD32 / NXP / ESP32
- RTOS shell
- USB 虚拟串口
- PC 代理程序

都建议沿用这套决策。

## 12. 一份可直接照着走的移植清单

如果要在新平台上重来一次，最推荐按下面的顺序执行：

1. 先定义运行方式、输入来源、标准库范围和错误恢复策略
2. 先把 PicoC 内核最小集合编译通过
3. 先实现 `PlatformPutc()`，确认欢迎信息和错误输出可见
4. 再实现最小输入接口，但由宿主掌控输入缓冲和状态机
5. 统一脚本执行入口，优先走 `PicocParse()`
6. 先跑通 `REPL`，再补整文件上传执行
7. 先保留 `mini-stdlib`，再逐步补头文件和平台扩展库
8. 最小移植阶段不绑定文件系统
9. 通过 `PlatformExit()` 和 `setjmp/longjmp` 打通错误恢复
10. 基础功能稳定后，再接断点、单步、变量查看和上位机调试

如果严格按这条路径推进，返工通常最少，调试成本也最低。
